# AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 포트폴리오

## 프로젝트 소개

이 프로젝트는 공개된 DCO 직무 정보와 교육용 샘플 데이터를 바탕으로
DCO 직무의 기본 용어, 교육용 로그 분석, 장애 상황 문서화와 인수인계 흐름을 학습한 결과물입니다.

생성형 AI를 단순히 답을 생성하는 도구로 사용하지 않고,
용어 정리, 문서 구조화, 로그 분석, Python 코드 작성, 보고서 초안 작성과 결과 검증을 돕는 도구로 활용했습니다.

이 프로젝트는 실제 AWS 내부 시스템, 실제 운영 로그, 실제 Ticket 또는 실제 데이터센터 장비를 대상으로 수행한 프로젝트가 아닙니다.

## 생성형 AI 활용 원칙

* 생성형 AI의 답변은 최종 정답이 아니라 검토가 필요한 초안으로 사용했습니다.
* AI가 작성한 장비명, 시간, Ticket ID와 사건 내용을 원본 로그와 비교했습니다.
* 로그에서 직접 확인되는 사실과 가능한 추정을 구분했습니다.
* 로그만으로 확인할 수 없는 Root Cause는 확정하지 않았습니다.
* Ticket 우선순위는 AI가 대신 결정하지 않고, 학생이 근거를 비교하여 직접 판단했습니다.
* 실제 계정정보, 실제 장비정보, 실제 Ticket과 실제 AWS 내부정보를 AI에 입력하지 않았습니다.
* 모든 로그, 장비명, Ticket과 IP 주소는 교육용 샘플 데이터로 구성했습니다.

## 배운 점

1. 생성형 AI의 한계 인지 및 교차 검증 능력 체득

 - AI를 활용해 로그 파싱 스크립트나 보고서 초안을 작성할 때, 대소문자 구분 누락이나 멀티라인 오류 등의 데이터 왜곡(환각 현상)이 발생할 수 있음을 배웠습니다. 이를 해결하기 위해 터미널에서 grep -i | wc -l 등의 수동 명령어로 숫자를 직접 대조하는 교차 검증(Cross Verification) 프로세스의 중요성을 깨달았습니다.

2. 장애 데이터 분석 시 사실(Fact)과 추정(Inference)의 엄격한 분리

 - 객관적인 장애 조치 보고서를 작성하기 위해서는 로그에 명시된 '확인된 사실'과 분석가의 '기술적 추정'을 철저히 분리해야 함을 배웠습니다. 특히, 패치 코드 교체 후 서비스가 정상화되었다는 사실만으로 장애의 최초 원인(Root Cause)을 성급하게 단정 짓지 않고, "~할 가능성"과 같은 조심스러운 표현을 사용하는 법을 익혔습니다.

3. 심각도(Severity)와 우선순위(Priority)의 명확한 차이 이해

 - 시스템 로그에 찍히는 기술적 위험도인 '심각도'와 실제 엔지니어가 처리해야 하는 '우선순위'는 다르다는 것을 배웠습니다. 멀티 티켓 상황 발생 시 단순 심각도에만 의존하지 않고, 현재 서비스 영향도, 반복·지속 여부, 잠재적 영향 범위, 복구 상태 및 정보 공백을 입체적으로 고려하여 조치 순서를 정립하는 triage(우선순위 분류) 역량을 길렀습니다.

4. 데이터센터 운영(DCO) 협업 및 문서화의 본질 파악

 - SOP를 체크리스트 형태로 시각화하여 휴먼 에러를 방지하는 법을 배웠으며, 장애 상황을 체계적으로 기록하는 Incident Report와 교대 근무 시 정보 공백 없는 인수인계를 위한 Shift Handover Memo 작성법을 실습했습니다. 이를 통해 다음 담당자 및 후속 기술 팀이 즉시 업무를 이어받을 수 있도록 시간과 현상을 명확하게 문서화하는 소통 능력이 DCO 직무의 핵심임을 깨달았습니다.

## 프로젝트 범위와 제한

이 저장소는 DCO 인턴십 직무를 이해하기 위한 학습형 포트폴리오입니다.

* 실제 AWS DCO 업무 수행 능력을 증명하는 자료가 아닙니다.
* 실제 데이터센터 운영 절차나 내부 SOP를 포함하지 않습니다.
* 실제 장애 대응, 장비 교체 또는 운영 자동화를 수행하지 않습니다.
* 공개 직무 맥락과 교육용 샘플 데이터를 이용한 학습 결과만 포함합니다.
